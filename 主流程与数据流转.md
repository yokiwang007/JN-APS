## 二、各阶段详细流程与数据流转

### 阶段1: 订单接收

#### 2.1.1 流程说明
从ERP/OMS系统接收客户订单,创建销售订单记录。这是整个APS系统的数据源头。

#### 2.1.2 数据流转

**输入数据源**: ERP/OMS系统

**数据对象**: `erp_sales_order` (销售订单主表)

**数据流向**:
```
ERP系统
  ↓ (订单数据)
创建 erp_sales_order 记录
  ↓
订单进入拆解队列
```

#### 2.1.3 数据对象详解

##### erp_sales_order (销售订单主表)

**作用**: 存储客户订单的基本信息,是整个生产流程的顶层业务对象,用于对账和最终发货。

**核心字段**:
```
order_id (PK)           - 订单号,唯一标识
customer_name           - 客户名称
due_date                - 承诺交期(客户要求的交货日期)
total_amount            - 订单总金额
priority                - 优先级(1-5,数字越大优先级越高)
fulfillment_rule        - 齐套规则:full(全齐套)/partial(可分批发货)
status                  - 订单状态(待拆解、已排程、生产中、已完成)
created_at              - 创建时间
updated_at              - 更新时间
```

**数据示例**:
```json
{
  "order_id": "ORD20260403001",
  "customer_name": "张三",
  "due_date": "2026-04-20",
  "total_amount": 15000.00,
  "priority": 3,
  "fulfillment_rule": "full",
  "status": "待拆解",
  "created_at": "2026-04-03T10:00:00",
  "updated_at": "2026-04-03T10:00:00"
}
```

**生命周期**:
1. 创建: 从ERP同步订单时创建
2. 更新: 订单状态变化时更新(已拆解→已排程→生产中→已完成)
3. 归档: 订单完成后归档

**关联关系**:
- 1:N 关联 aps_production_order (一个订单可拆分为多个生产子订单)
- 1:1 关联 mes_sorting_slot (一个订单对应一个齐套货位)

### 阶段2: 订单拆解

#### 2.2.1 流程说明
将销售订单按产品BOM拆解为底层零件(板件/五金件),并根据零件的工艺属性(材质、厚度、封边模式等)自动匹配工艺路线,生成"生产子订单(工艺包)"。这是从"销售订单"转换为"生产订单"的关键转换步骤。

#### 2.2.2 数据流转

**输入数据源**:
- erp_sales_order (销售订单主表)
- 产品BOM (物料清单,来自设计软件如酷家乐/三维家)
- 工艺路线库 (ProcessMapping)

**数据对象**:
- cad_bom_part (底层零件明细表)
- aps_production_order (生产子订单表)

**数据流向**:
```
erp_sales_order (销售订单)
  ↓
解析BOM,生成 cad_bom_part (零件明细)
  ↓
应用工艺映射规则
  ↓
按工艺路线分组,生成 aps_production_order (生产子订单)
  ↓
子订单进入排程队列
```

#### 2.2.3 数据对象详解

##### cad_bom_part (底层零件明细表)

**作用**: 存储订单的最小生产单元(每块板件、每个五金件),是最细颗粒度的数据对象,由前端设计软件解析生成。

**核心字段**:
```
part_id (PK)            - 零件ID,唯一标识
order_id (FK)           - 所属销售订单
part_type               - 零件类型(柜体板、门板、背板、五金件等)
length                  - 长度
width                   - 宽度
thickness               - 厚度
material                - 材质(颗粒板、多层板、密度板等)
color                   - 花色(子午灰、白橡木、樱桃木等)
edge_banding            - 封边模式(四边封直边、无封边等)
barcode                 - 条码号(用于生产追踪)
area                    - 面积
process_route           - 工艺路线(自动匹配)
```

**数据示例**:
```json
{
  "part_id": "PART20260403001001",
  "order_id": "ORD20260403001",
  "part_type": "柜体板",
  "length": 600,
  "width": 400,
  "thickness": 18,
  "material": "颗粒板",
  "color": "子午灰",
  "edge_banding": "四边封直边",
  "barcode": "BC20260403001001",
  "area": "0.2400",
  "process_route": "标准柜体线"
}
```

**生命周期**:
1. 创建: 订单拆解时创建
2. 分配: 揉单合并时分配到批次
3. 追踪: 生产执行时追踪工序状态
4. 完成: 完工后标记为已完成

**关联关系**:
- N:1 关联 erp_sales_order (多个零件属于一个订单)
- N:1 关联 aps_production_order (多个零件属于一个生产子订单)
- N:1 关联 aps_batch_detail (多个零件属于一个批次)

##### aps_production_order (生产子订单表)【核心转换对象】

**作用**: 这是APS真正排程的输入对象。系统根据BOM中的材质和加工动作,自动将一个销售订单拆成多个子订单,每个子订单具有唯一的工艺路线,对应一个"生产线"(工作中心)。

**核心字段**:
```
production_order_id (PK) - 子订单ID,唯一标识
order_id (FK)            - 所属销售订单
process_route            - 工艺路线(标准柜体线/异形门板线/吸塑线等)
total_part_count         - 总件数
material                 - 材质
thickness                - 厚度
color                    - 花色
edge_banding             - 封边模式
planned_date             - 计划生产日期(粗排程后填充)
line_id                  - 分配产线(工作中心,粗排程后填充)
status                   - 状态(待排程、已排程、已合并、生产中、已完成)
created_at               - 创建时间
updated_at               - 更新时间
```

**数据示例**:
```json
{
  "production_order_id": "PO20260403001001",
  "order_id": "ORD20260403001",
  "process_route": "标准柜体线",
  "total_part_count": 30,
  "material": "颗粒板",
  "thickness": 18,
  "color": "子午灰",
  "edge_banding": "四边封直边",
  "planned_date": null,
  "line_id": null,
  "status": "待排程",
  "created_at": "2026-04-03T10:30:00",
  "updated_at": "2026-04-03T10:30:00"
}
```

**生命周期**:
1. 创建: 订单拆解时创建
2. 排程: 粗排程时填充 planned_date 和 line_id
3. 合并: 揉单合并时关联到批次
4. 完成: 生产完成后标记为已完成

**关联关系**:
- N:1 关联 erp_sales_order (多个生产子订单属于一个销售订单)
- 1:N 关联 cad_bom_part (一个生产子订单包含多个零件)
- 1:1 关联 aps_schedule_task (一个生产子订单对应一个排程任务)

#### 2.2.4 工艺映射规则

**规则配置示例**:
```
IF 材质=颗粒板 AND 厚度=18mm AND 形状=矩形
  THEN process_route = "标准柜体线"

IF 材质=密度板 AND 工序包含(CNC铣型)
  THEN process_route = "吸塑门板线"

IF 材质=多层板 AND 厚度=9mm
  THEN process_route = "背板线"
```

### 阶段3: 粗排程

#### 2.3.1 流程说明
基于交期和产能,为每个生产子订单确定计划生产日期和产线分配。此阶段不生成批次,只确定每个子订单大致何时、在哪个产线生产。采用"倒排计划"方法,从交期倒推开工日期。

#### 2.3.2 数据流转

**输入数据源**:
- aps_production_order (生产子订单)
- sys_work_center (工作中心/虚拟产线)
- 策略配置(倒排计划规则、优先级规则、产能负荷规则、齐套窗口规则)

**数据对象**:
- sys_work_center (工作中心表)
- aps_schedule_task (排程任务表)

**数据流向**:
```
aps_production_order (生产子订单)
  ↓
读取 sys_work_center (产线产能)
  ↓
应用倒排计划规则
  最迟开工日 = 交期 - 齐套缓冲期 - 生产周期
  ↓
分配产线和计划日期
  ↓
创建/更新 aps_schedule_task (排程任务)
  ↓
更新 aps_production_order 的 planned_date 和 line_id
  ↓
子订单进入揉单队列
```

#### 2.3.3 数据对象详解

##### sys_work_center (工作中心/虚拟产线表)

**作用**: 定义各个生产线的产能信息,是排程时的资源约束。这里的"产线"是逻辑上的工作中心,不是具体的物理设备。

**核心字段**:
```
line_id (PK)            - 产线ID,唯一标识
line_name               - 产线名称(柜体A线、门板线、吸塑线等)
line_type               - 产线类型(开料线、封边线、钻孔线等)
daily_capacity          - 日产能(部件件数)
daily_capacity_area     - 日产能(平方米)
work_center_ids         - 关联的设备工作中心ID列表(逗号分隔)
status                  - 状态(正常、维护、停用)
```

**数据示例**:
```json
{
  "line_id": "LINE001",
  "line_name": "柜体A线",
  "line_type": "标准柜体线",
  "daily_capacity": 500,
  "daily_capacity_area": 150.0,
  "work_center_ids": "WC001,WC002,WC003",
  "status": "正常"
}
```

**生命周期**:
- 静态数据,由系统管理员配置
- 定期更新产能数据(如设备升级后)

**关联关系**:
- 1:N 关联 aps_schedule_task (一个产线可安排多个排程任务)
- 1:N 关联 aps_merge_batch (一个产线可生成多个批次)

##### aps_schedule_task (排程任务表)【阶段1的持久化输出】

**作用**: 固化粗排程结果。一旦写入,车间的宏观计划就定了。此表记录每个生产子订单的计划日期和产线分配。

**核心字段**:
```
task_id (PK)            - 任务ID,唯一标识
production_order_id (FK) - 关联生产子订单ID
planned_start_date      - 计划开工日期
planned_end_date        - 计划完工日期
line_id                 - 分配的产线编码
priority                - 优先级(继承自销售订单或根据规则调整)
schedule_status         - 排产状态(已排定/已揉单)
created_at              - 创建时间
updated_at              - 更新时间
```

**数据示例**:
```json
{
  "task_id": "TASK20260403001001",
  "production_order_id": "PO20260403001001",
  "planned_start_date": "2026-04-15",
  "planned_end_date": "2026-04-17",
  "line_id": "LINE001",
  "priority": 3,
  "schedule_status": "已排定",
  "created_at": "2026-04-03T14:00:00",
  "updated_at": "2026-04-03T14:00:00"
}
```

**生命周期**:
1. 创建: 粗排程时创建
2. 更新: 手动调整排程时更新
3. 完成: 揉单合并后更新 schedule_status = "已揉单"

**关联关系**:
- 1:1 关联 aps_production_order (一个任务对应一个生产子订单)
- N:1 关联 sys_work_center (多个任务属于一个产线)

#### 2.3.4 倒排计划示例

假设订单交期为 2026-04-20,齐套缓冲期为 2 天,柜体线生产周期为 2 天:

```
最迟完工日 = 交期 - 齐套缓冲期
           = 2026-04-20 - 2天
           = 2026-04-18

最迟开工日 = 最迟完工日 - 生产周期
           = 2026-04-18 - 2天
           = 2026-04-16

因此:
- planned_start_date = 2026-04-16
- planned_end_date = 2026-04-18
```

### 阶段4: 揉单合并

#### 2.4.1 流程说明
将分配在同一天、同一产线的任务打碎,为了"提高出材率"按物理属性(材质、花色、厚度、封边模式)重新揉捏成"生产批次"。这是从"订单维度"转换为"批次维度"的关键步骤,也是定制家具行业提高材料利用率的核心环节。

#### 2.4.2 数据流转

**输入数据源**:
- aps_schedule_task (排程任务,状态为"已排定")
- 策略配置(揉单时间窗规则、交期红线策略、批次规模规则、合并原则)

**数据对象**:
- aps_merge_batch (生产批次主表)
- aps_batch_detail (批次明细映射表)

**数据流向**:
```
aps_schedule_task (排程任务)
  ↓
按产线和日期筛选待揉单任务
  ↓
按材质、花色、厚度、封边模式分组
  ↓
应用揉单时间窗规则和交期红线策略
  ↓
检查批次规模是否在最优区间
  ↓
生成 aps_merge_batch (批次主表)
  ↓
生成 aps_batch_detail (批次明细,关联零件)
  ↓
更新 aps_schedule_task 的 schedule_status = "已揉单"
  ↓
批次进入细排程队列
```

#### 2.4.3 数据对象详解

##### aps_merge_batch (生产批次主表)【阶段2的持久化输出】

**作用**: 这是一个全新的业务对象。下发给开料机和封边机的,不再是订单,而是这个批次号。批次是车间执行的基本单元。

**核心字段**:
```
batch_id (PK)           - 批次号,唯一标识(如 B20260415-LINE001-001)
line_id (FK)            - 生产产线
production_date         - 计划生产日期
material                - 材质
thickness               - 厚度
color                   - 花色
total_part_count        - 批次总部件数
merge_rule              - 使用的合并规则(板材利用率优先/设备利用率优先)
utilization_rate        - 板材利用率(开料优化后填充)
optimize_status         - 优化状态(待优化/已优化)
status                  - 状态(待排程、已排程、生产中、已完成)
created_at              - 创建时间
updated_at              - 更新时间
```

**数据示例**:
```json
{
  "batch_id": "B20260415-LINE001-001",
  "line_id": "LINE001",
  "production_date": "2026-04-15",
  "material": "颗粒板",
  "thickness": 18,
  "color": "子午灰",
  "total_part_count": 60,
  "merge_rule": "板材利用率优先",
  "utilization_rate": null,
  "optimize_status": "待优化",
  "status": "待排程",
  "created_at": "2026-04-03T16:00:00",
  "updated_at": "2026-04-03T16:00:00"
}
```

**生命周期**:
1. 创建: 揉单合并时创建
2. 优化: 细排程时调用开料优化算法,填充 utilization_rate
3. 下发: 下发至MES时更新 status = "生产中"
4. 完成: 生产完成后更新 status = "已完成"

**关联关系**:
- N:1 关联 sys_work_center (多个批次属于一个产线)
- 1:N 关联 aps_batch_detail (一个批次包含多个零件)
- 1:1 关联 cam_cutting_pattern (一个批次对应一个排版图)

##### aps_batch_detail (批次明细映射表)

**作用**: 承上启下。记录这个批次里到底包含了哪些订单的哪些零件。这是从"批次"找回"订单"的关键关联表。

**核心字段**:
```
id (PK)                 - 自增ID
batch_id (FK)           - 批次ID
production_order_id (FK) - 生产子订单ID
part_id (FK)            - 零件ID
```

**数据示例**:
```json
{
  "id": 1,
  "batch_id": "B20260415-LINE001-001",
  "production_order_id": "PO20260403001001",
  "part_id": "PART20260403001001"
}
```

**生命周期**:
- 随 aps_merge_batch 的创建而创建
- 随 aps_merge_batch 的删除而删除

**关联关系**:
- N:1 关联 aps_merge_batch (多条明细属于一个批次)
- N:1 关联 aps_production_order (多条明细属于一个生产子订单)
- N:1 关联 cad_bom_part (多条明细属于一个零件)

#### 2.4.4 揉单时间窗规则示例

**配置**: 最大提前1天,最大延后1天

**场景**:
- 订单A: 计划日期 4月15日, 交期 4月20日
- 订单B: 计划日期 4月16日, 交期 4月21日
- 两者材质、花色、厚度相同

**揉单过程**:
1. 检查时间窗: B的计划日期(4月16日)与A的计划日期(4月15日)相差1天,在时间窗范围内
2. 检查交期: B提前到4月15日生产,完工时间仍满足交期要求(4月21日)
3. 系统将A和B合并为一个批次,安排在4月15日生产

#### 2.4.5 交期红线熔断示例

**场景**:
- 订单C: 计划日期 4月15日(最迟开工日), 交期 4月20日
- 材质: 稀有花色, 只有10件
- 时间窗范围内找不到同花色订单

**熔断过程**:
1. 系统检测到订单C已达到最迟开工日
2. 即使只有10件,利用率极低,触发交期红线熔断
3. 强制生成批次,标记为[加急/强制下发]
4. 提示计划员"该批次利用率低,但为保证交期强制下发"

### 阶段5: 细排程

#### 2.5.1 流程说明
对批次进行精确排程,确定具体开始时间、结束时间、设备分配,并调用开料优化算法生成排版图。此阶段的输出是车间可以直接执行的生产指令。

#### 2.5.2 数据流转

**输入数据源**:
- aps_merge_batch (待排程的批次)
- aps_batch_detail (批次明细,包含零件信息)
- sys_work_center (产线设备信息)
- 策略配置(排程算法、设备产能、换型时间)

**数据对象**:
- cam_cutting_pattern (排版图/锯切图表)
- 排程明细(可存储在 aps_merge_batch 或单独的排程明细表)

**数据流向**:
```
aps_merge_batch (待排程批次)
  ↓
读取 aps_batch_detail (批次零件信息)
  ↓
执行排程算法,确定精确时间
  ↓
调用开料优化算法(第三方库如海迅、星辉)
  ↓
生成排版图,计算利用率
  ↓
创建 cam_cutting_pattern (排版图记录)
  ↓
更新 aps_merge_batch 的 utilization_rate 和 optimize_status
  ↓
更新 aps_merge_batch 的 status = "已排程"
  ↓
批次进入生产执行队列
```

#### 2.5.3 数据对象详解

##### cam_cutting_pattern (排版图/锯切图表)

**作用**: 存储开料优化结果,包括所需大板数量、排版图路径、利用率等。由第三方优化算法(如海迅、星辉)算完后回传保存。

**核心字段**:
```
pattern_id (PK)         - 图纸ID,唯一标识
batch_id (FK)           - 关联批次号
required_board_count    - 所需原大板数量
board_specification     - 大板规格(如 1220x2440mm 18mm)
utilization_rate        - 预计利用率(如 92.5%)
waste_rate              - 废料率
image_path              - 优化图路径(存储排版图文件)
created_at              - 创建时间
```

**数据示例**:
```json
{
  "pattern_id": "PATTERN2026041500001",
  "batch_id": "B20260415-LINE001-001",
  "required_board_count": 10,
  "board_specification": "1220x2440mm 18mm",
  "utilization_rate": 92.5,
  "waste_rate": 7.5,
  "image_path": "/patterns/B20260415-LINE001-001.png",
  "created_at": "2026-04-03T18:00:00"
}
```

**生命周期**:
1. 创建: 细排程时创建
2. 使用: 生产执行时读取排版图
3. 归档: 生产完成后归档

**关联关系**:
- 1:1 关联 aps_merge_batch (一个批次对应一个排版图)

#### 2.5.4 排程明细

**说明**: 排程明细可以存储在 aps_merge_batch 表中(增加 start_time 和 end_time 字段),也可以创建单独的排程明细表。建议创建单独的表,便于管理复杂的排程信息。

**可选字段**:
```
schedule_id (PK)        - 排程ID
batch_id (FK)           - 批次ID
process_name            - 工序名称(开料、封边、钻孔等)
start_datetime          - 开始时间
end_datetime            - 结束时间
duration_minutes        - 耗时(分钟)
equipment_id            - 设备ID
```

### 阶段6: 生产执行

#### 2.6.1 流程说明
将批次数据下发至MES,生成生产工单,车间执行生产并反馈进度。通过扫码追踪每块板件在各工序的状态。

#### 2.6.2 数据流转

**输入数据源**:
- aps_merge_batch (已排程的批次)
- cam_cutting_pattern (排版图)
- aps_batch_detail (批次明细)

**数据对象**:
- mes_wip_tracking (在制品工序扫码表)
- 生产工单(ProductionOrder,可与MES系统集成)

**数据流向**:
```
aps_merge_batch (已排程批次)
  ↓
下发至MES,生成生产工单
  ↓
车间执行生产
  ↓
工序完成时扫码
  ↓
创建/更新 mes_wip_tracking (工序追踪记录)
  ↓
更新零件状态
  ↓
批次所有零件完成后,更新 aps_merge_batch status = "已完成"
  ↓
更新 aps_production_order status = "已完成"
  ↓
触发齐套检查
```

#### 2.6.3 数据对象详解

##### mes_wip_tracking (在制品工序扫码表)

**作用**: 追踪每块板件在各个工序的完成情况,实现生产过程的可视化管理。

**核心字段**:
```
tracking_id (PK)        - 追踪ID,唯一标识
part_id (FK)            - 零件ID
batch_id (FK)           - 所属批次
current_process         - 当前工序(已开料/已封边/已打孔/已完成)
operation_time          - 操作时间
operator                - 操作人
equipment_id            - 设备ID
status                  - 状态(进行中/已完成/异常)
```

**数据示例**:
```json
{
  "tracking_id": "TRACK2026041500001",
  "part_id": "PART20260403001001",
  "batch_id": "B20260415-LINE001-001",
  "current_process": "已封边",
  "operation_time": "2026-04-15T14:30:00",
  "operator": "操作员001",
  "equipment_id": "EQ001",
  "status": "已完成"
}
```

**生命周期**:
1. 创建: 零件进入第一个工序时创建
2. 更新: 每完成一个工序时更新
3. 完成: 所有工序完成后标记为"已完成"

**关联关系**:
- N:1 关联 cad_bom_part (多条追踪记录属于一个零件)
- N:1 关联 aps_merge_batch (多条追踪记录属于一个批次)

##### 生产工单(ProductionOrder)

**说明**: 生产工单通常由MES系统管理,APS系统可以创建工单记录或将批次信息传递给MES。

**核心字段**:
```
production_order_id (PK) - 工单号
batch_id (FK)            - 关联批次
work_center_id           - 加工工作中心
quantity                 - 加工数量
start_time               - 实际开始时间
end_time                 - 实际结束时间
status                   - 状态(待生产/生产中/已完成)
```

### 阶段7: 成品齐套与发货

#### 2.7.1 流程说明
监控订单齐套状态,管理发货流程。根据齐套规则,判断订单是否允许分批发货,并生成发货单。

#### 2.7.2 数据流转

**输入数据源**:
- erp_sales_order (销售订单)
- aps_production_order (生产子订单)
- mes_wip_tracking (在制品追踪)
- 策略配置(齐套规则、分批发货策略)

**数据对象**:
- mes_sorting_slot (齐套分拣货位表)
- 发货单(Shipment)

**数据流向**:
```
mes_wip_tracking (工序追踪记录)
  ↓
按零件ID聚合,统计订单完成进度
  ↓
查询 erp_sales_order 的 fulfillment_rule
  ↓
判断是否满足齐套条件
  ↓
创建/更新 mes_sorting_slot (齐套货位)
  ↓
满足齐套条件时,生成发货建议
  ↓
生成发货单
  ↓
更新 erp_sales_order status = "已完成"
```

#### 2.7.3 数据对象详解

##### mes_sorting_slot (齐套分拣货位表)【解决排程时间差的神器】

**作用**: 物理车间的缓存库映射。记录每个订单的齐套进度,管理分拣货位。这是解决"不同子订单在不同日期完工"导致等待问题的关键机制。

**核心字段**:
```
slot_id (PK)            - 货架槽位ID,唯一标识
order_id (FK)           - 当前分配的销售订单号
total_parts             - 总部件数
completed_parts         - 已完成部件数
completion_progress     - 齐套进度(如 "98/100")
completion_rate         - 齐套率(如 98%)
estimated_completion_date - 预计齐套日期
waiting_days            - 等待天数(已完工部件的等待时间)
sort_status             - 分拣状态(待齐套/部分齐套/已齐套/已发货)
created_at              - 创建时间
updated_at              - 更新时间
```

**数据示例**:
```json
{
  "slot_id": "SLOT001",
  "order_id": "ORD20260403001",
  "total_parts": 100,
  "completed_parts": 98,
  "completion_progress": "98/100",
  "completion_rate": 98.0,
  "estimated_completion_date": "2026-04-18",
  "waiting_days": 2,
  "sort_status": "部分齐套",
  "created_at": "2026-04-15T10:00:00",
  "updated_at": "2026-04-17T16:00:00"
}
```

**生命周期**:
1. 创建: 订单的第一个零件完工时创建
2. 更新: 每有零件完工时更新进度
3. 齐套: 所有零件完工时更新 sort_status = "已齐套"
4. 发货: 发货后更新 sort_status = "已发货"

**关联关系**:
- 1:1 关联 erp_sales_order (一个订单对应一个齐套货位)

##### 发货单(Shipment)

**说明**: 发货单通常由ERP/OMS系统管理,APS系统可以生成发货建议或将齐套信息传递给ERP。

**核心字段**:
```
shipment_id (PK)        - 发货单号
order_id (FK)           - 关联订单
shipment_date           - 发货日期
shipment_type           - 发货类型(全齐套/分批发货)
parts_list              - 部件清单
status                  - 状态(待发货/已发货)
```

#### 2.7.4 齐套检查示例

**场景**:
- 订单ORD20260403001包含3个生产子订单:
  - PO001(柜体): 30件, 预计4月15日完工
  - PO002(门板): 20件, 预计4月17日完工
  - PO003(五金): 10件, 已采购入库
- 齐套规则: full(全齐套)

**齐套过程**:
1. 4月15日: PO001完工, mes_sorting_slot 更新为 "30/60", sort_status = "部分齐套"
2. 4月17日: PO002完工, mes_sorting_slot 更新为 "50/60", sort_status = "部分齐套"
3. 4月18日: PO003入库, mes_sorting_slot 更新为 "60/60", sort_status = "已齐套"
4. 系统触发发货建议,生成发货单

## 四、关键数据流转场景

### 4.1 场景一: 单订单完整流转

**订单信息**:
- 订单号: ORD20260403001
- 客户: 张三
- 交期: 2026-04-20
- 齐套规则: full(全齐套)
- 产品: 衣柜(包含柜体板30件、门板20件、五金件10件)

**完整数据流转**:

1. **阶段1: 订单接收**
   - 创建 erp_sales_order:
     ```json
     {
       "order_id": "ORD20260403001",
       "customer_name": "张三",
       "due_date": "2026-04-20",
       "fulfillment_rule": "full",
       "status": "待拆解"
     }
     ```

2. **阶段2: 订单拆解**
   - 创建 cad_bom_part (60条记录):
     - 30条柜体板记录
     - 20条门板记录
     - 10条五金件记录
   - 创建 aps_production_order (2条记录):
     ```json
     // 柜体子订单
     {
       "production_order_id": "PO001",
       "order_id": "ORD20260403001",
       "process_route": "标准柜体线",
       "total_part_count": 30,
       "material": "颗粒板",
       "status": "待排程"
     }
     // 门板子订单
     {
       "production_order_id": "PO002",
       "order_id": "ORD20260403001",
       "process_route": "吸塑门板线",
       "total_part_count": 20,
       "material": "密度板",
       "status": "待排程"
     }
     ```

3. **阶段3: 粗排程**
   - 倒排计划:
     - 柜体线生产周期2天,最迟完工日=4月20-2=4月18日,开工日=4月16日
     - 门板线生产周期5天,最迟完工日=4月20-2=4月18日,开工日=4月13日
   - 创建 aps_schedule_task (2条记录):
     ```json
     // 柜体任务
     {
       "task_id": "TASK001",
       "production_order_id": "PO001",
       "planned_start_date": "2026-04-16",
       "planned_end_date": "2026-04-18",
       "line_id": "LINE001",
       "schedule_status": "已排定"
     }
     // 门板任务
     {
       "task_id": "TASK002",
       "production_order_id": "PO002",
       "planned_start_date": "2026-04-13",
       "planned_end_date": "2026-04-18",
       "line_id": "LINE002",
       "schedule_status": "已排定"
     }
     ```
   - 更新 aps_production_order:
     - PO001: planned_date="2026-04-16", line_id="LINE001", status="已排程"
     - PO002: planned_date="2026-04-13", line_id="LINE002", status="已排程"

4. **阶段4: 揉单合并**
   - 假设柜体线4月16日有多个订单,PO001与其他订单合并
   - 创建 aps_merge_batch:
     ```json
     {
       "batch_id": "B20260416-LINE001-001",
       "line_id": "LINE001",
       "production_date": "2026-04-16",
       "material": "颗粒板",
       "total_part_count": 60,  // 包含PO001的30件+其他订单的30件
       "status": "待排程"
     }
     ```
   - 创建 aps_batch_detail (60条记录,关联30条cad_bom_part)
   - 假设门板线4月13日只有PO002,单独成批
   - 创建 aps_merge_batch:
     ```json
     {
       "batch_id": "B20260413-LINE002-001",
       "line_id": "LINE002",
       "production_date": "2026-04-13",
       "material": "密度板",
       "total_part_count": 20,
       "status": "待排程"
     }
     ```
   - 更新 aps_schedule_task.schedule_status = "已揉单"

5. **阶段5: 细排程**
   - 调用开料优化算法
   - 创建 cam_cutting_pattern (2条记录):
     ```json
     {
       "pattern_id": "PATTERN001",
       "batch_id": "B20260416-LINE001-001",
       "required_board_count": 12,
       "utilization_rate": 92.5
     }
     {
       "pattern_id": "PATTERN002",
       "batch_id": "B20260413-LINE002-001",
       "required_board_count": 5,
       "utilization_rate": 88.0
     }
     ```
   - 更新 aps_merge_batch:
     - utilization_rate = 92.5 / 88.0
     - optimize_status = "已优化"
     - status = "已排程"

6. **阶段6: 生产执行**
   - 下发至MES,生成生产工单

     - 更新 aps_merge_batch.status = "已完成"
     - 更新 aps_production_order.status = "已完成"

### 4.2 场景二: 多订单揉单合并

**订单信息**:
- 订单A: 柜体,30件,计划日期4月15日,材质=颗粒板,花色=子午灰
- 订单B: 柜体,35件,计划日期4月16日,材质=颗粒板,花色=子午灰
- 订单C: 柜体,40件,计划日期4月15日,材质=多层板,花色=子午灰

**揉单过程**:

1. **筛选待揉单任务**
   - 柜体线4月15日: 订单A(30件) + 订单C(40件)
   - 柜体线4月16日: 订单B(35件)

2. **按材质花色分组**
   - 组1: 颗粒板+子午灰, 订单A(30件)
   - 组2: 多层板+子午灰, 订单C(40件)
   - 组3: 颗粒板+子午灰, 订单B(35件)

3. **应用时间窗规则**(最大提前1天,最大延后1天)
   - 订单B的计划日期(4月16日)与订单A的计划日期(4月15日)相差1天,在时间窗范围内
   - 检查交期: 订单B提前1天生产,仍满足交期要求

4. **合并组1和组3**

## 六、总结

### 6.1 数据流转核心要点

1. **数据对象清晰化**: 每个阶段都有独立的物理表承载业务对象,数据流转清晰
2. **关系明确化**: 通过外键约束保证数据一致性,关联关系清晰
3. **状态可控化**: 每个数据对象都有明确的生命周期和状态机
4. **责任单一化**: 每个数据对象都有明确的职责,避免职责混乱

### 6.2 关键数据转换点

1. **订单拆解**: 从"销售订单"转换为"生产子订单"
   - 数据源: erp_sales_order → aps_production_order
   - 转换逻辑: 按工艺路线分组

2. **粗排程**: 从"生产子订单"转换为"排程任务"
   - 数据源: aps_production_order → aps_schedule_task
   - 转换逻辑: 倒排计划,分配产线和日期

3. **揉单合并**: 从"排程任务"转换为"生产批次"
   - 数据源: aps_schedule_task → aps_merge_batch
   - 转换逻辑: 按材质花色分组,应用时间窗规则

4. **细排程**: 从"生产批次"转换为"排版图"
   - 数据源: aps_merge_batch → cam_cutting_pattern
   - 转换逻辑: 开料优化算法

5. **成品齐套**: 从"零件完成"转换为"订单齐套"
   - 数据源: mes_wip_tracking → mes_sorting_slot
   - 转换逻辑: 按订单聚合,计算齐套进度

### 6.3 数据一致性保证

1. **外键约束**: 通过数据库外键保证关联数据的一致性
2. **事务控制**: 关键操作使用事务保证原子性
3. **状态机**: 每个数据对象都有明确的状态流转规则
4. **数据校验**: 关键节点进行数据完整性校验
